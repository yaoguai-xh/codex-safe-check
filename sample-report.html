一、症状摘要

客户描述：安装后新开终端仍提示 codex: command not found。在另一个终端中偶尔能看到旧版本，但登录流程无法打开浏览器。

二、已确认的脱敏证据

检查项结果说明
系统与架构macOS / arm64只保留系统家族和架构，不记录用户名或主机名。
Codex CLI当前 shell 未找到；另一个 shell 显示旧版本更像 PATH 或多 Node 环境问题。
Node/npmNode 存在；npm 全局目录存在需要确认当前 shell 是否包含 npm 全局 bin。
Git/curl均存在基础工具链不是主要嫌疑。
网络可达性安装地址可达；API 主机 HTTPS 检查失败安装和登录可能是两个问题。
代理/证书检测到代理变量和自定义 CA 变量存在只记录“存在”，不记录变量值。

三、最可能原因

  1. PATH 没有指向当前 npm 全局 bin。 证据：一个 shell 找不到 codex，另一个 shell 能看到旧版本，说明终端初始化文件或 Node 管理器存在分歧。置信度：高。
  2. 旧版本 Codex 或旧 npm 全局目录残留。 证据：不同终端返回不一致。置信度：中。
  3. 代理/自定义 CA 影响登录或 HTTPS 连接。 证据：代理和 CA 变量存在，API 主机 HTTPS 检查失败。置信度：中。

四、按顺序操作

每次只改变一个因素。完成一步后先验证，避免同时改 PATH、代理和证书导致无法回溯。

步骤 1：确认当前 shell 看到的命令位置

command -v codex
npm bin -g
  • 用途：判断 codex 是否在当前 shell 的 PATH 中。
  • 预期结果：两个输出中至少能看到 npm 全局 bin 与 Codex 可执行文件关联。
  • 如果失败：只反馈“command -v 是否为空”和“npm bin -g 是否有输出”，不要发送完整路径。

步骤 2：统一当前 shell 的 Node/npm 来源

如果你使用 nvm、fnm、asdf、Homebrew 或系统 Node，只保留当前终端实际使用的一套。不要同时改多个 shell 配置文件。

  • 用途：避免一个终端安装，另一个终端运行旧版本。
  • 预期结果：新开终端后 node --version、npm --version 和 codex --version 来源一致。
  • 停止条件：如果新开终端仍不一致，先不要处理代理，继续回到 PATH。

步骤 3：在客户本人设备上重新触发登录

codex login
  • 用途：验证命令可执行后，确认登录流程是否能启动。
  • 预期结果：浏览器登录由客户本人完成，CLI 回到可用状态。
  • 如果失败：只反馈一行脱敏错误摘要，不发送认证文件或完整登录输出。

步骤 4：如果仍是 HTTPS/证书错误，再处理代理或 CA

  • 用途：只在 PATH 和登录入口确认后，才处理网络层。
  • 预期结果：客户确认代理软件、公司网络或自定义 CA 是否是预期配置。
  • 停止条件：涉及公司设备、TLS 拦截或根证书时，交给管理员确认，不建议关闭证书校验。

五、验证标准

  • codex --version 在新终端中稳定返回同一版本。
  • 客户本人完成登录流程，不向任何人发送凭据或认证文件。
  • 如果网络仍失败，错误已被归类为代理/证书/端点可达性，而不是安装问题。

六、仍需补充的信息

  1. 新开终端后 command -v codex 是否为空。
  2. codex --version 是否稳定返回同一版本。
  3. 登录失败时的一行脱敏错误摘要。

七、边界说明

本报告针对收到的脱敏环境证据提供排错建议，不保证第三方服务、账号资格、组织策略或特定网络环境一定恢复。涉及登录、付款、账号和组织权限的操作，必须由客户本人完成。
